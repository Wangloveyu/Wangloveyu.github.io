  • 【操作】系统调用的整个过程是什么(原理)

    1. 系统调用通过系统调用指令触发异常，使得系统从用户态转变为内核态(即将CPU从用户程序让渡给系统内核程序执行相关操作)
    2. 执行系统调用后，系统首先保存执行系统调用的位置，之后将保存当前进程的相关信息(通用寄存器等)，之后进行中断服务程序寻址，找到中断服务程序的地址并执行
    3. 中断服务程序执行完毕后
    4. 内核程序恢复进程信息，并从内核态切换回用户态
    5. 切换回用户态后执行系统调用的下一条指令

  • 【操作】中断处理的过程

    1. 关闭中断，避免在处理时有更高级别的中断导致现场保存不完整
    2. 保存断点
    3. 中断服务程序寻址
    4. 保存现场
    5. 开启中断
    6. 执行中断服务程序
    7. 关闭中断，保证在恢复现场时不被中断
    8. 恢复现场
    9. 开启中断
    10. 注意：前三步骤由硬件完成，后面的步骤由中断服务程序完成

  • 【前端】npm管理包的方式或者具体过程

    1. 没有lock文件

      • 分析依赖关系，这是因为一个包会依赖其他的包，并且多个包之间会产生相同依赖的情况；
      • 从registry仓库中下载压缩包
      • 获取到压缩包后会对压缩包进行缓存
      • 将压缩包解压到项目的node_modules文件夹中，并生成package-lock文件

    2. 有lock文件

      • 检查lock中包的版本是否和package.json中是否一致
      • 不一致，那么会重新构建依赖关系，直接会走没有lock文件的流程
      • 一致的情况下，会去优先查找缓存
      • 没有找到，会从registry仓库下载，直接会走没有lock文件的流程
      • 查找到，会获取缓存中的压缩文件，并且将压缩文件解压到node_modules文件夹中

  • 【前端】http缓存

    • http缓存分为http强缓存和http协商缓存

    • http强缓存

      • 当命中强缓存时，客户端不会再请求服务器，直接从缓存中读取内容，并返回状态码200

    • 协商缓存

      • 浏览器向服务器询问缓存相关的信息，如果服务器返回304未修改，则表示之前的资源仍可以使用
      • 相关的请求头
      • last-modified和etag

    • 启发式缓存

      • 即使没有给出Cache-Control，浏览器一般也会将http进行缓存，缓存的有效时间为响应头中Date的值 - Last-Modified的值的差除以10
      • 因此建议给出Cache-Control，避免启发式缓存

    • 原理

      • 服务器在响应头中设置expires或cache-control，expires是一个资源过期时间，而cache-control通过max-age参数来指定一个资源的有效持续时间，当服务器要再次请求资源时，如果仍在cache-control的max-age指定的时间范围内，则直接使用缓存缓存资源

        • expires是对cache-control的向下兼容，因此现在可以不设置expires，只设置cache-control

      • 如果想强制验证缓存，则服务器可以设置Cache-Control: no-cache，这样浏览器会每次发送请求向服务器验证缓存中的内容

        • 如果请求的资源已更新，客户端将收到 200 OK 响应，否则，如果请求的资源尚未更新，则会收到 304 Not Modified 响应

      • 如果不想使用缓存，则服务器可以使用Cache-Control: no-store

    • 注意：http缓存是浏览器的后退/前进缓存的基础

  • 【前端】浏览器是如何渲染页面的

    1. HTML被HTML解析器解析成DOM树

    2. CSS被CSS解析器解析成CSS规则树

    3. 浏览器会将CSS规则树附着在DOM树上，并结合两者生成渲染树Render Tree

    4. 生成布局（flow），浏览器通过解析计算出每一个渲染树节点的位置和大小，在屏幕上画出渲染树的所有节点

    5. 将布局绘制（paint）在屏幕上，显示出整个页面

    6. 注意：浏览器本质上是以图层的方式进行绘制的，因此如果某个节点重绘的话，需要重绘整个图层，但是这个开销小于重排的开销

  • 【前端】diff算法

    1. diff 算法是一种通过同层的树节点进行比较的高效算法

      其有两个特点：

      • 比较只会在同层级进行, 不会跨层级比较
      • 在diff比较的过程中，循环从两边向中间比较

    2. 流程

      1. 比较一下新旧节点是不是同一个节点(可以通过比较选择器或key来判断)

      2. 如果不是同一个节点，则先以旧节点为基准插入新节点，然后再删除旧节点

      3. 若是同一个节点则需要进一步比较

        1. 完全相同，不做处理

        2. 新节点内容为文本，直接替换

        3. 新节点有子节点：

          1. 若老节点没有子元素，则直接清空老节点，将新节点的子元素插入即可
          2. 若老节点有子元素则重复上述的更新策略

  • 【前端】cookie，sessionStorage和localStorage的区别

    • 写入方式：cookie是由服务器端写入的，而SessionStorage、 LocalStorage都是由前端写入的
    • 生命周期：cookie的生命周期是由服务器端在写入的时候就设置好的，LocalStorage是写入就一直存在，除非手动清除，SessionStorage是页面关闭的时候就会自动清除。
    • 存储大小：cookie的存储空间比较小大概4KB，SessionStorage、 LocalStorage存储空间比较大，大概5M。
    • 数据共享：三者数据共享都遵循同源原则，SessionStorage还限制必须是同一个页面。
    • 发送请求是否携带：在前端给后端发送请求的时候会自动携带Cookie中的数据，但是SessionStorage、 LocalStorage不会。
    • 应用场景：Cookie一般用于存储登录验证信息SessionID或者token，LocalStorage常用于存储不易变动的数据，减轻服务器的压力，SessionStorage可以用来检测用户是否是刷新进入页面，如音乐播放器恢复播放进度条的功能。

  • 【前端】https工作流程

    • https与http相比主要是多了加密流程，加密的方式是非对称加密

    • 首先服务器生成公私钥对，公钥发给CA机构，私钥自己保管

    • CA机构使用自己的公私钥对，利用私钥对服务器发来的公钥进行签名生成CA证书

    • CA机构将生成的CA证书交给服务器

    • CA机构使用的公钥会被内置到浏览器作为CA根证书

    • https建立连接的流程

      1. 客户端使用三次握手发起连接
      2. 服务器将CA证书返回给客户端，证书中包含了服务器的公钥，颁发机构，有效期等
      3. 客户端根据浏览器内置的CA根证书中的CA机构的公钥验证CA证书的合法性
      4. 之后客户端生成随机对称密钥并通过服务器的公钥进行加密发送给服务器
      5. 服务器通过私钥进行解密，之后双方都可以通过客户端生成的随机密钥进行http通信

  • 【前端】React的Context什么时候会触发子组件重新渲染

    • 当没有正确处理好Context的使用位置的时候，Context会导致所有的子组件重新渲染，例如如果直接在某一个组件中使用Context.Provide，则会导致Context.Provide标签中的所有子组件都会在Provide的值改变的时候被重新渲染，因为这种写法下，标签中的所有子组件都是在Context.Provide创建的过程中一起创建的，而不是独立的
    • 当Context.Provide被抽离出来成为一个新组件，然后子组件通过props.children的形式传递的时候，Provide的值被改变只会影响后续使用到其值的组件，因为props.children在这里已经在传递进来前就已经被创建好了
    • 因此结论很明显，Context没有使用好的时候，其值被改变，那么所有的子组件都会被重新渲染，当Context用好的时候，只有使用到其值的组件才会被重新渲染

  • 【前端】script标签放在head里和放在body底部有什么区别？

    1. 浏览器是从上到下解析HTML的。

    2. 放在head里的js代码，会在body解析之前被解析；放在body底部的js代码，会在整个页面加载完成之后解析。

  • 【前端】什么是作用域链？

    • 作用域，即变量（变量作用域又称上下文）和函数生效（能被访问）的区域或集合

    • 换句话说，作用域决定了代码区块中变量和其他资源的可见性 我们一般将作用域分成：

      • 全局作用域：任何不在函数中或是大括号中声明的变量，都是在全局作用域下，全局作用域下声明的变量可以在程序的任意位置访问
      • 函数作用域：函数作用域也叫局部作用域，如果一个变量是在函数内部声明的它就在一个函数作用域下面。这些变量只能在函数内部访问，不能在函数以外去访问
      • 块级作用域：ES6引入了let和const关键字,和var关键字不同，在大括号中使用let和const声明的变量存在于块级作用域中。在大括号之外不能访问这些变量

    • 什么是作用域链：当在Javascript中使用一个变量的时候，首先Javascript引擎会尝试在当前作用域下去寻找该变量，如果没找到，再到它的上层作用域寻找，以此类推直到找到该变量或是已经到了全局作用域

    • 如果在全局作用域里仍然找不到该变量，它就会在全局范围内隐式声明该变量(非严格模式下)或是直接报错

      • 然后从各个根对象开始遍历，把不是垃圾的节点改成1

      • 清理所有标记为0的垃圾，销毁并回收它们所占用的内存空间

      • 最后，把所有内存中对象标记修改为0，等待下一轮垃圾回收

      • 缺点

        • 由于是从根对象(全局对象)开始查找，对于那些无法从根对象查询到的对象都将被清除

        • 回收后会形成内存碎片，影响后面申请大的连续内存空间

          • 可以通过标记整理算法进行优化，即将活跃对象移动到内存的一端，然后清理掉边界内存即可

    • 引用计数

      • 声明一个变量，被引用时计数+1

      • 同一个值被赋予另一个变量时，计数+1

      • 保存该引用的变量被其他值覆盖时，计数-1

      • 计数为0时回收内存

      • 缺点

        • 循环引用

  • 【前端】html文档渲染过程，css文件和js文件的下载，是否会阻塞渲染？

    • CSS阻塞

      • css 文件的下载和解析不会影响 DOM 的解析，但是会阻塞 DOM 的渲染。因为 CSSOM Tree 要和 DOM Tree 合成 Render Tree 才能绘制页面。

      • css 文件的下载不会阻塞前面的 js 脚本执行。（所以在需要提前执行不操作 dom 元素的 js 时，不妨把 js 放到 css 文件之前。）

    • js阻塞 ​ - js 文件的下载和解析会阻塞 GUI 渲染进程，也就是会阻塞 DOM 和 CSS 的解析和渲染 ​ - js 文件没下载并解析完成之前，后续的 HTML 和 CSS 无法解析 ​ - js 文件的下载不会阻塞前面 HTML 和 CSS 的解析（当js放在body底部时）

  • 【前端】如何实现一个扫描二维码登录PC网站的需求？

    具体流程：生成二维码 -> 扫描二维码 -> 状态确认

    1. 生成二维码

    • PC端向服务端发起请求，告诉服务端，我要生成用户登录的二维码，并且把PC端设备信息也传递给服务端
    • 服务端收到请求后，它生成二维码ID，并将二维码ID与PC端设备信息进行绑定
    • 然后把二维码ID返回给PC端
    • PC端收到二维码ID后，生成二维码(二维码中肯定包含了ID)
    • 为了及时知道二维码的状态，客户端在展现二维码后，PC端不断的轮询服务端，比如每隔一秒就轮询一次，请求服务端告诉当前二维码的状态及相关信息，或者直接使用websocket，等待在服务端完成登录后进行通知

    2. 扫描二维码

    • 用户用手机去扫描PC端的二维码，通过二维码内容取到其中的二维码ID
    • 再调用服务端API将移动端的身份信息与二维码ID一起发送给服务端
    • 服务端接收到后，它可以将身份信息与二维码ID进行绑定，生成临时token。然后返回给手机端
    • 因为PC端一直在轮询二维码状态，所以这时候二维码状态发生了改变，它就可以在界面上把二维码状态更新为已扫描

    3. 状态确认

    • 手机端在接收到临时token后会弹出确认登录界面，用户点击确认时，手机端携带临时token用来调用服务端的接口，告诉服务端，我已经确认
    • 服务端收到确认后，根据二维码ID绑定的设备信息与账号信息，生成用户PC端登录的token
    • 这时候PC端的轮询接口，它就可以得知二维码的状态已经变成了"已确认"。并且从服务端可以获取到用户登录的token
    • 到这里，登录就成功了，后端PC端就可以用token去访问服务端的资源了

      const currying = function (fn, ...args) {
        // fn需要的参数个数
        const len = fn.length;
        // 返回一个函数接收剩余参数
        return function (...params) {
          // 拼接已经接收和新接收的参数列表
          let _args = [...args, ...params];
          // 如果已经接收的参数个数还不够，继续返回一个新函数接收剩余参数
          if (_args.length < len) {
            return currying.apply(this, fn, _args);
          }
          // 参数全部接收完调用原函数
          return fn.apply(this, _args);
        };
      };

  • 【前端】React Fiber主要都做了些什么？

    • React Fiber主要用于React内部进行调度，使得React内部的批处理操作变为可中断操作，从而避免了当React内部在进行复杂的更新操作时导致浏览器造成卡顿
    • React Fiber的原理是利用了时间片的原理，可以类比操作系统，它使用浏览器渲染每一帧剩下的时间进行计算，使得浏览器渲染不会被阻塞
    • Fiber主要利用了RequestAnimationFrame(高优先级任务)和requestIdleCallback(低优先级任务)
